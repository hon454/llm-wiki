# Karpathy의 LLM Wiki

[[andrej-karpathy]]가 제안한 LLM 기반 개인 지식 베이스 구축 패턴.

## 핵심 아이디어

기존 [[rag]] 방식은 질의할 때마다 원본 문서에서 관련 청크를 검색하여 답변을 생성한다. 지식이 축적되지 않고 매번 처음부터 재발견해야 한다.

LLM Wiki는 다르다. LLM이 **점진적으로 영구 위키를 구축하고 유지**한다. 새 소스가 추가되면 단순 인덱싱이 아니라, 기존 위키에 통합한다 — 엔티티 페이지 업데이트, 주제 요약 수정, 모순 감지, 종합 강화. 지식은 한 번 컴파일되고 계속 최신 상태를 유지한다.

> **위키는 영구적이고 복리로 성장하는 산출물이다.** 교차 참조는 이미 존재하고, 모순은 이미 표시되었으며, 종합은 읽은 모든 것을 반영한다.

## 3계층 아키텍처

1. **Raw sources** — 원본 소스 문서 모음 (불변, LLM이 읽기만 함)
2. **Wiki** — LLM이 생성·관리하는 마크다운 파일 디렉토리 (요약, 엔티티, 개념, 비교, 종합)
3. **Schema** — 위키 구조, 규칙, 워크플로를 정의하는 설정 파일 (CLAUDE.md, AGENTS.md 등)

## 3가지 연산

- **Ingest**: 새 소스를 읽고, 요약하고, 기존 위키 페이지들을 업데이트 (한 소스가 10~15개 페이지에 영향)
- **Query**: 위키를 검색하여 답변 생성. 좋은 답변은 다시 위키에 저장하여 지식이 복리로 축적
- **Lint**: 위키 건강 검진 — 모순, 고아 페이지, 누락된 교차 참조 등 점검

## 인덱싱과 로깅

- **index.md**: 콘텐츠 중심 카탈로그. 카테고리별 정리. LLM이 질의 시 먼저 읽고 관련 페이지로 이동
- **log.md**: 시간순 작업 이력. `grep` 파싱 가능한 형식으로 기록

## 활용 사례

- 개인 성장 추적 (일기, 기사, 팟캐스트 노트)
- 연구 (논문, 보고서 축적)
- 독서 (챕터별 정리, 등장인물·주제 위키)
- 비즈니스 (Slack, 회의록, 고객 통화 기반 내부 위키)
- 경쟁 분석, 실사, 여행 계획 등

## 도구

- [[obsidian]]을 뷰어로 사용 (그래프 뷰, 백링크, Dataview, Web Clipper)
- 규모가 커지면 qmd 같은 로컬 검색 엔진 추가 가능
- Git으로 버전 관리, 브랜칭, 협업

## 철학적 배경

[[memex]](Vannevar Bush, 1945)의 비전과 맥을 같이한다 — 문서 간 연상 경로가 있는 개인 큐레이션 지식 저장소. Bush가 풀지 못한 유지보수 문제를 LLM이 해결한다.
